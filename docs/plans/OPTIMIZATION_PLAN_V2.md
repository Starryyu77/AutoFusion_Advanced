# AutoFusion 优化实验方案 (协作版)

## 一、核心目标
让NAS自动发现的架构在**性能**和**效率**上均优于人工设计(FiLM)

**成功标准**:
- MMMU准确率 > 46% (当前NAS ~33%, FiLM ~46%)
- FLOPs < 6.29M (当前NAS ~50M, FiLM 6.29M)

---

## 二、你需要做的核心工作

### 任务1: 改进评估器 (最关键)
**问题**: 当前AI2D+3epochs无法预测复杂任务性能

**行动**:
- 修改 `experiment/evaluators/real_data_evaluator.py`
- 添加配置: dataset='mmmu', train_epochs=10, num_shots=32
- 添加早停: 验证集3epoch不提升即停止
- 添加时间限制: 单个架构训练不超过5分钟

**验证标准**: 跑通一次完整搜索迭代(生成→评估→得Reward)

---

### 任务2: 改进Reward函数
**问题**: 当前效率权重太低(0.5)，导致架构臃肿

**行动**:
- 修改 `experiment/base/reward.py`
- 提升效率权重: efficiency_weight=1.5 (原为0.5)
- 添加FLOPs硬约束: >10M直接reject
- 添加FLOPs指数惩罚: reward *= exp(-flops/20M)

**验证标准**: 生成的架构平均FLOPs < 15M

---

### 任务3: 运行改进版架构搜索
**行动**:
- 使用新评估器+新Reward运行Phase 4搜索
- 配置: 200 iterations, population=50
- 保存Top 20架构

**验证标准**: Best Reward > 0.9, 平均FLOPs < 10M

---

### 任务4: 全面评估(New-NAS vs 所有Baseline)
**行动**:
- 对新发现的Top 5架构运行E1+E2
- 对比对象: 5个旧Baseline + 8个旧NAS + 新NAS
- 生成对比表格和帕累托前沿图

**验证标准**: 至少1个新NAS在MMMU上超越FiLM

## 五、检查点

### Checkpoint 1
**条件**: 任务1完成 (新评估器跑通)
**决策**:
- 如果MMMU评估时间 < 5分钟: 继续
- 如果时间太长: 改用VSR或AI2D+20ep

### Checkpoint 2
**条件**: 任务3完成 (新NAS架构生成)
**决策**:
- 如果平均FLOPs < 10M: 进入任务4
- 如果还是太臃肿: 调整Reward硬约束

### Checkpoint 3
**条件**: 任务4完成 (全面对比)
**决策**:
- 如果新NAS > FiLM: 准备论文
- 如果接近: 补充消融实验
- 如果仍落后: 分析原因，决定是否需要再一轮优化

## 六、风险应对

| 风险 | 应对方案 |
|------|----------|
| MMMU评估太慢 | 使用早停+时间限制，或换VSR数据集 |
| SOTA复现困难 | 优先选有开源代码的，或简化实现 |
| 新NAS仍不如FiLM | 分析FiLM设计模式，加入搜索空间 |
